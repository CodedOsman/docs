## Overview

The Pico detects rising sanitation air-quality risk, escalates local alerts, and allows a user to acknowledge the buzzer while keeping visible warning states active.

Alert systems are less useful when they either stay silent until conditions are severe or continue buzzing even after a user has seen the warning.

A Pico 2 W prototype with an MQ-135 gas sensor, acknowledgement button, two LEDs, and a buzzer.

Multi-level alerting, acknowledgement behavior, and innovation grounded in practical sanitation response design.

### Project Story

**Advanced Project**: This advanced project is designed to help learners move beyond basic wiring and coding into complete system thinking. The learner should build the prototype, test each subsystem, validate the data, explain the design decisions, and propose improvements for real-world deployment.

Sanitation teams need alerts that are noticeable, explainable, and not overly disruptive once someone has responded. This prototype helps students design a more thoughtful local warning system by separating visible status from the buzzer acknowledgement path.

## Step-by-Step Assembly

1. Connect the MQ-135 analog output to GPIO 26 through a safe ADC path.
2. Wire the acknowledgement button between GPIO 14 and 3.3V so the input goes high when pressed.
3. Connect the green LED to GPIO 15 through a 220 ohm resistor and the red LED to GPIO 16 through another 220 ohm resistor.
4. Connect the buzzer positive pin to GPIO 17 and its negative pin to ground.
5. Warm up the MQ-135 fully before choosing the environmental thresholds.

## Full Project Code

After completing and checking the circuit connections, open Thonny IDE, copy and paste this code into a new file or upload the project file to the Raspberry Pi Pico 2 W, then run it from Thonny.

Upload and run this code after the individual tests work correctly.

```python
from machine import ADC, Pin
import time

GAS_PIN = 26
BUTTON_PIN = 14
GREEN_LED_PIN = 15
RED_LED_PIN = 16
BUZZER_PIN = 17
WARN_THRESHOLD = 18000
CRITICAL_THRESHOLD = 30000

mq135 = ADC(GAS_PIN)
button = Pin(BUTTON_PIN, Pin.IN, Pin.PULL_DOWN)
green_led = Pin(GREEN_LED_PIN, Pin.OUT)
red_led = Pin(RED_LED_PIN, Pin.OUT)
buzzer = Pin(BUZZER_PIN, Pin.OUT)
acknowledged = False
last_button = 0

print('Smart sanitation innovation system ready')

while True:
    gas_value = mq135.read_u16()
    button_now = button.value()

    if gas_value >= CRITICAL_THRESHOLD:
        state = 'CRITICAL'
    elif gas_value >= WARN_THRESHOLD:
        state = 'WARNING'
    else:
        state = 'NORMAL'
        acknowledged = False

    if button_now == 1 and last_button == 0 and state != 'NORMAL':
        acknowledged = True
        print('Alert acknowledged')
    last_button = button_now

    green_led.value(1 if state == 'NORMAL' else 0)
    red_led.value(1 if state in ('WARNING', 'CRITICAL') else 0)
    buzzer.value(1 if state in ('WARNING', 'CRITICAL') and not acknowledged else 0)

    print('Gas:{} State:{} Ack:{}'.format(gas_value, state, acknowledged))
    time.sleep(1)
```

## How the Code Works

| Code Section | What It Does | Why It Matters | What to Modify During Testing |
|--------------|--------------|----------------|------------------------------|
| Alert classification | Maps the gas reading into NORMAL, WARNING, or CRITICAL. | Clear levels are easier for users to understand than raw sensor values alone. | Set the thresholds only after sensor warm-up and baseline measurement. |
| Acknowledgement logic | Lets the user silence the buzzer while keeping the visible warning active. | This is the main innovation idea of the project and makes the alert more practical to use. | Check the button state during testing if acknowledgement seems unreliable. |
| Output control | Drives the green LED, red LED, and buzzer according to the current state. | Separate visible and audible behavior gives a more realistic local response design. | If a buzzer stays on unexpectedly, print the current state and acknowledgement flag. |

## Expected Result

The green LED stays on during normal conditions and the red LED stays off.

When the gas reading rises above the warning threshold, the red LED and buzzer activate.

Pressing the button silences the buzzer while the red LED continues to show that the warning condition still exists.

### Validation Checks

- **Normal condition test**: confirm the green LED is on and the red LED and buzzer are off after warm-up.
- **Warning condition test**: raise the gas reading above the warning threshold and confirm the red LED and buzzer turn on.
- **Critical condition test**: raise the gas reading further and confirm the system reports CRITICAL.
- **Acknowledgement test**: press the button during a warning or critical condition and confirm only the buzzer turns off.
- **Reset test**: let the reading return to normal and confirm acknowledgement clears automatically for the next alert event.

### Deployment and Limitations

- This prototype can be used for local warning demonstrations and early field trials.
- Before deployment, it needs enclosure protection, calibration records, and a clear response procedure.
- Alert systems should always be treated as decision-support tools unless they are certified for the application.

## Troubleshooting

| Problem | Possible Cause | Solution |
|---------|----------------|----------|
| The buzzer never sounds | The buzzer wiring is wrong or the acknowledgement flag was set already | Test the buzzer directly from a short script and confirm acknowledgement resets when the state returns to normal. |
| The buzzer never stops | The button wiring is wrong or the state never leaves warning | Print the button value during testing and verify the alert threshold against the warmed-up baseline. |
| The red LED is always on | The warning threshold is too low or the MQ-135 is still warming up | Allow full warm-up and raise the threshold after measuring the room baseline. |
| The button does not acknowledge the alert | The button input never goes high | Test the button alone and verify it is wired between GPIO 14 and 3.3V. |
