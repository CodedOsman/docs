## Overview

The Pico combines live flow rate and tank level to estimate how quickly stored water may run out.

Water planning is difficult when users know the current tank level but not how fast that water is being consumed.

A Pico 2 W prototype with a pulse flow sensor, water-level sensor, and OLED used as a local water-intelligence dashboard.

Data fusion, remaining-runtime estimation, and honest interpretation of local water availability.

### Project Story

**Advanced Project**: This advanced project is designed to help learners move beyond basic wiring and coding into complete system thinking. The learner should build the prototype, test each subsystem, validate the data, explain the design decisions, and propose improvements for real-world deployment.

Water decisions improve when users can compare what is left in storage with how quickly it is being consumed. This prototype helps students estimate whether current use is sustainable without claiming exact forecasting accuracy.

## Full Project Code

After completing and checking the circuit connections, open Thonny IDE, copy and paste this code into a new file or upload the project file to the Raspberry Pi Pico 2 W, then run it from Thonny.

```python
from machine import ADC, I2C, Pin
import time

try:
    from ssd1306 import SSD1306_I2C
    HAS_OLED = True
except ImportError:
    HAS_OLED = False

FLOW_PIN = 14
LEVEL_PIN = 27
I2C_SDA_PIN = 20
I2C_SCL_PIN = 21
LEVEL_EMPTY = 5000
LEVEL_FULL = 50000
SAMPLE_WINDOW_S = 5
TANK_CAPACITY_L = 20.0

pulse_count = 0
flow = Pin(FLOW_PIN, Pin.IN, Pin.PULL_UP)
level = ADC(LEVEL_PIN)
i2c = I2C(0, sda=Pin(I2C_SDA_PIN), scl=Pin(I2C_SCL_PIN), freq=400000)
if HAS_OLED:
    oled = SSD1306_I2C(128, 64, i2c)


def count_pulse(pin):
    global pulse_count
    pulse_count += 1


def clamp(value, low, high):
    return max(low, min(high, value))


def level_percent(raw):
    span = LEVEL_FULL - LEVEL_EMPTY
    if span <= 0:
        return 0
    pct = int((raw - LEVEL_EMPTY) * 100 / span)
    return clamp(pct, 0, 100)


def flow_lpm_from_pulses(pulses, seconds):
    if seconds <= 0:
        return 0.0
    return (pulses / seconds) / 7.5


flow.irq(trigger=Pin.IRQ_FALLING, handler=count_pulse)
print('Water intelligence system ready')

while True:
    pulse_count = 0
    start = time.ticks_ms()
    while time.ticks_diff(time.ticks_ms(), start) < SAMPLE_WINDOW_S * 1000:
        time.sleep_ms(100)

    flow_lpm = flow_lpm_from_pulses(pulse_count, SAMPLE_WINDOW_S)
    tank_pct = level_percent(level.read_u16())
    remaining_liters = (tank_pct / 100) * TANK_CAPACITY_L
    if flow_lpm > 0:
        remaining_minutes = (remaining_liters / flow_lpm) * 60
    else:
        remaining_minutes = 9999

    if tank_pct < 20 and remaining_minutes < 30:
        status = 'CRITICAL'
    elif tank_pct < 40 or remaining_minutes < 60:
        status = 'WATCH'
    else:
        status = 'NORMAL'

    print('Flow:{:.2f}L/min Tank:{}% Rem:{:.0f}min Status:{}'.format(flow_lpm, tank_pct, remaining_minutes, status))

    if HAS_OLED:
        oled.fill(0)
        oled.text('Water IQ', 0, 0)
        oled.text('Flow:{:.1f}L'.format(flow_lpm), 0, 16)
        oled.text('Tank:{}%'.format(tank_pct), 0, 30)
        oled.text(status[:16], 0, 46)
        oled.show()
```

## How the Code Works

| Code Section | What It Does | Why It Matters | What to Modify During Testing |
|--------------|--------------|----------------|------------------------------|
| Dual sensing | Combines flow-rate measurement with tank-level measurement | Water intelligence depends on both usage and availability | Validate each sensor separately before trusting the combined estimate |
| Remaining-runtime estimate | Calculates how long the current tank level might last at the present flow rate | This gives the user a more actionable picture than tank level alone | Remember that the estimate is only as accurate as the current flow assumption |
| Status logic | Maps the combined level and runtime estimate into NORMAL, WATCH, or CRITICAL | This turns the fused data into a planning decision | Adjust the watch and critical limits after testing your own scenario |
| OLED dashboard | Shows the local water-intelligence summary | The dashboard helps students compare fused data with a practical decision label | Verify ssd1306.py and I2C wiring if the display is blank |

### Validation Checks

- **Normal condition test**: confirm the system stays in its safe or idle state under baseline conditions
- **Warning condition test**: move the input close to the chosen threshold and confirm the transition is sensible
- **Critical condition test**: trigger the strongest alert or control state and confirm the output response is correct
- **Calibration test**: compare the sensor or timing against a known reference or repeated trial
- **Limitation test**: deliberately create an awkward or noisy condition and note how the prototype behaves

### Deployment and Limitations

- This prototype works well for classroom demonstrations and local monitoring pilots
- Before deployment, it needs calibration records, protective mounting, and regular validation checks
- Display-only systems still depend on correct sensor placement and trained users

## Troubleshooting

| Problem | Possible Cause | Solution |
|---------|----------------|----------|
| Remaining minutes look unrealistic | The tank capacity or flow calibration is wrong | Check the pulse-to-flow calibration and confirm the tank capacity assumption |
| Tank percentage never changes | The analog level path is wrong or the calibration values are unrealistic | Print the raw level value and re-measure empty and full references |
| Flow is always zero | The pulse signal is not reaching GPIO 14 | Print the pulse count during a known flow event and recheck the signal path |
| OLED is blank | The display library or I2C wiring is wrong | Confirm ssd1306.py is present and recheck GPIO 20 and GPIO 21 |
