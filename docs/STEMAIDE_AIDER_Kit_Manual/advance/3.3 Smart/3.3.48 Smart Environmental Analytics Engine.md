## Overview

The Pico tracks environmental readings over time, computes rolling averages, and flags unusual changes.

Environmental measurements are hard to interpret when users see only the newest reading and not the recent trend behind it.

A Pico 2 W prototype with an MQ-5 Liquefied Petroleum/Methane Gas Sensor, BME280, and OLED analytics dashboard.

Rolling averages, anomaly-style detection, and honest local analytics rather than a cloud analytics engine.

### Project Story

**Advanced Project**: This advanced project is designed to help learners move beyond basic wiring and coding into complete system thinking. The learner should build the prototype, test each subsystem, validate the data, explain the design decisions, and propose improvements for real-world deployment.

Environmental monitoring often becomes more useful when teams can spot change over time instead of checking one reading in isolation. This prototype helps students build a local analytics engine that summarizes recent conditions and highlights unusual readings without overclaiming a large remote platform.

---

## Learning Objectives

- Collect environmental data into a short rolling history
- Compute rolling averages for gas, temperature, and humidity
- Flag anomaly-style changes when a new reading differs strongly from the recent average
- Use a local dashboard to compare a current reading with its trend context
- Discuss the difference between local analytics and a real data pipeline or cloud analytics engine
- Validate whether anomaly rules are too sensitive or too weak

### Project-Specific Safety Note

Gas sensor readings are for learning and prototype use only and should not be treated as certified safety measurements.

Many MQ sensor modules use a 5V heater and can output up to 5V on the analog pin. Use a resistor divider or a 3.3V-safe conditioning board before connecting the analog output to the Pico ADC.

Treat the anomaly labels as prototype indicators only and verify them against real test conditions.

## Full Project Code

After completing and checking the circuit connections, open Thonny IDE, copy and paste this code into a new file or upload the project file to the Raspberry Pi Pico 2 W, then run it from Thonny.

Upload and run this code after the individual tests work correctly.

```python
from machine import ADC, I2C, Pin
import bme280
import time

try:
    from ssd1306 import SSD1306_I2C
    HAS_OLED = True
except ImportError:
    HAS_OLED = False

GAS_PIN = 26
BME_SDA_PIN = 20
BME_SCL_PIN = 21
I2C_SDA_PIN = 20
I2C_SCL_PIN = 21
HISTORY_LENGTH = 6

mq5 = ADC(GAS_PIN)
i2c = I2C(0, sda=Pin(BME_SDA_PIN), scl=Pin(BME_SCL_PIN), freq=400000)
climate = bme280.BME280(i2c=i2c)
if HAS_OLED:
    oled = SSD1306_I2C(128, 64, i2c)

gas_history = []
temp_history = []
hum_history = []


def average(values):
    return sum(values) / len(values) if values else 0


print('Smart environmental analytics engine ready')

while True:
    gas_value = mq5.read_u16()
    try:
        values = climate.values
        temperature = float(values[0].replace('C', ''))
        pressure = float(values[1].replace('hPa', ''))
        humidity = float(values[2].replace('%', ''))
    except Exception:
        temperature = 0
        pressure = 0
        humidity = 0

    gas_history.append(gas_value)
    temp_history.append(temperature)
    hum_history.append(humidity)
    gas_history = gas_history[-HISTORY_LENGTH:]
    temp_history = temp_history[-HISTORY_LENGTH:]
    hum_history = hum_history[-HISTORY_LENGTH:]

    avg_gas = average(gas_history)
    avg_temp = average(temp_history)
    avg_hum = average(hum_history)

    if len(gas_history) >= 3 and (gas_value - avg_gas >= 6000 or temperature - avg_temp >= 4):
        label = 'ANOMALY'
    elif gas_value > avg_gas + 2000 or temperature > avg_temp + 1:
        label = 'RISING'
    else:
        label = 'STABLE'

    print('Gas:{} AvgG:{:.0f} Temp:{} AvgT:{:.1f} Hum:{} AvgH:{:.1f} Label:{}'.format(gas_value, avg_gas, temperature, avg_temp, humidity, avg_hum, label))

    if HAS_OLED:
        oled.fill(0)
        oled.text('Env Analytics', 0, 0)
        oled.text('G:{} A:{:.0f}'.format(gas_value, avg_gas), 0, 16)
        oled.text('T:{} H:{}'.format(temperature, humidity), 0, 32)
        oled.text(label[:16], 0, 50)
        oled.show()

    time.sleep(3)
```

## How the Code Works

| Code Section | What It Does | Why It Matters | What to Modify During Testing |
|--------------|--------------|----------------|------------------------------|
| History buffers | Stores recent gas, temperature, and humidity readings. | Analytics depends on recent context rather than only the latest number. | Increase or decrease HISTORY_LENGTH to change how stable the averages feel. |
| Rolling averages | Calculates the average baseline for each environmental signal. | Averages give the user a trend reference for the latest reading. | Watch how the average changes during gradual versus sudden test conditions. |
| Anomaly logic | Compares the newest reading against the recent average and labels the result. | This is the local analytics behavior promised by the project title. | Tune the anomaly deltas after repeated tests in your own room. |

## Expected Result

The dashboard shows current environmental readings together with a trend or anomaly label.

Gradual change tends to produce RISING while sharp change produces ANOMALY after the history window fills.

The rolling averages make it easier to explain why the label changed.

### Validation Checks

- **Normal condition test**: let the room remain stable and confirm the label becomes STABLE.
- **Trend test**: create a gradual temperature or gas increase and confirm the label changes to RISING.
- **Anomaly test**: create a sharper change and confirm the label changes to ANOMALY.
- **Calibration test**: warm up the MQ-5 before collecting any baseline values.
- **Sensitivity test**: adjust the history length and compare how quickly the labels change.

### Deployment and Limitations

- This prototype works well for classroom demonstrations and local monitoring pilots.
- Before deployment, it needs calibration records, protective mounting, and regular validation checks.
- Display-only systems still depend on correct sensor placement and trained users.

## Troubleshooting

| Problem | Possible Cause | Solution |
|---------|----------------|----------|
| The label is always anomaly | The thresholds are too sensitive or the gas sensor has not stabilized | Allow more warm-up time and increase the anomaly threshold after observing a baseline. |
| The averages never change | The history is not updating or the room conditions are too static | Print the history length and vary the environment more clearly during testing. |
| Climate values stay at zero | The BME280 read failed | Test the BME280 separately and shorten the wiring if needed. |
| OLED is blank | The library or I2C wiring is wrong | Confirm ssd1306.py is present and verify GPIO 20 and GPIO 21. |
