## Overview

The Pico fuses air quality, climate, and light readings into one combined environmental condition label.

Single-sensor interpretation can be misleading when environmental quality depends on several factors at once.

A Pico 2 W prototype with an MQ-135 gas sensor, DHT22, light sensor, and OLED fusion dashboard.

Multi-sensor fusion, weighted scoring, and transparent interpretation of combined environmental data.

### Project Story

**Advanced Project**: This advanced project is designed to help learners move beyond basic wiring and coding into complete system thinking. The learner should build the prototype, test each subsystem, validate the data, explain the design decisions, and propose improvements for real-world deployment.

Real environments are shaped by more than one measurement at a time. This prototype helps students combine air quality, heat, humidity, and light information into one fused condition label while still keeping the decision explainable.

### Project-Specific Safety Note

Gas sensor readings are for learning and prototype use only and should not be treated as certified safety measurements.

Many MQ sensor modules use a 5V heater and can output up to 5V on the analog pin. Use a resistor divider or a 3.3V-safe conditioning board before connecting the analog output to the Pico ADC.

Treat the fused label as a prototype interpretation and not as a certified environmental assessment.

## Step-by-Step Assembly

2. Build the light sensor analog path and connect it to GPIO 27.
3. Connect the DHT22 to Pico 3.3V, GND, and GPIO 15.
4. Connect the OLED to Pico 3.3V, GND, GPIO 20, and GPIO 21.

## Testing Individual Components

Before running the full project, test each subsystem separately. This makes it easier to find wiring, library, or logic problems before full integration.

1. **Hardware setup**: Assemble the Pico, sensor, indicator, and load wiring exactly as shown in the connection table before applying power.
2. **Test the input sensor**: Test the gas, light, and climate sensors separately so the fused score is built from trusted inputs.
3. **Test the output device**: Run an OLED hello-world test before adding the fusion logic.
4. **Test the decision logic**: Change one factor at a time and confirm the fused score responds for understandable reasons.
5. **Run the full system**: Run the full dashboard and compare the fused label with the contributing raw values.
6. **Validate the prototype**: Discuss whether any one sensor is dominating the fusion score too much.
7. **Save the project**: Save the validated program on the Pico as main.py and keep a copy on the computer for future edits.

## Full Project Code

After completing and checking the circuit connections, open Thonny IDE, copy and paste this code into a new file or upload the project file to the Raspberry Pi Pico 2 W, then run it from Thonny.

Upload and run this code after the individual tests work correctly.

```python
from machine import ADC, I2C, Pin
import dht
import time

try:
    from ssd1306 import SSD1306_I2C
    HAS_OLED = True
except ImportError:
    HAS_OLED = False

GAS_PIN = 26
LIGHT_PIN = 27
DHT_PIN = 15
I2C_SDA_PIN = 20
I2C_SCL_PIN = 21

mq135 = ADC(GAS_PIN)
light = ADC(LIGHT_PIN)
climate = dht.DHT22(Pin(DHT_PIN))
i2c = I2C(0, sda=Pin(I2C_SDA_PIN), scl=Pin(I2C_SCL_PIN), freq=400000)
if HAS_OLED:
    oled = SSD1306_I2C(128, 64, i2c)

print('Smart environmental data fusion ready')

while True:
    gas_value = mq135.read_u16()
    light_raw = light.read_u16()
    try:
        climate.measure()
        temperature = climate.temperature()
        humidity = climate.humidity()
    except OSError:
        temperature = 0
        humidity = 0

    score = 0
    if gas_value >= 18000:
        score += 2
    if temperature >= 30:
        score += 1
    if humidity >= 75:
        score += 1
    if light_raw < 18000:
        score += 1

    if score >= 4:
        label = 'STRESSED'
    elif score >= 2:
        label = 'WATCH'
    else:
        label = 'CLEAR'

    print('Gas:{} Light:{} Temp:{}C Hum:{}% Score:{} Label:{}'.format(gas_value, light_raw, temperature, humidity, score, label))

    if HAS_OLED:
        oled.fill(0)
        oled.text('Env Fusion', 0, 0)
        oled.text('G:{} L:{}'.format(gas_value, light_raw), 0, 18)
        oled.text('T:{} H:{}'.format(temperature, humidity), 0, 34)
        oled.text(label[:16], 0, 50)
        oled.show()

    time.sleep(2)
```

## How the Code Works

| Code Section | What It Does | Why It Matters | What to Modify During Testing |
|--------------|--------------|----------------|------------------------------|
| Weighted fusion score | Assigns points to several signals and combines them. | This is the core data-fusion behavior of the project. | Change one weight at a time so you can still explain the result. |
| Fused label mapping | Converts the score into CLEAR, WATCH, or STRESSED. | A clear label makes the fused condition easier to interpret quickly. | Tune the score boundaries after repeated trials. |
| OLED dashboard | Shows the sensor values and fused label together. | The user can compare the evidence with the final decision on one screen. | Verify ssd1306.py and I2C wiring if the display is blank. |

## Expected Result

The serial monitor reports the current reading or state clearly.

Subsystem behavior matches the thresholds, timing, or rules described in the document.

### Validation Checks

- **Normal condition test**: confirm the system stays in its safe or idle state under baseline conditions.
- **Warning condition test**: move the input close to the chosen threshold and confirm the transition is sensible.
- **Critical condition test**: trigger the strongest alert or control state and confirm the output response is correct.
- **Calibration test**: compare the sensor or timing against a known reference or repeated trial.
- **Limitation test**: deliberately create an awkward or noisy condition and note how the prototype behaves.

### Deployment and Limitations

- This prototype works well for classroom demonstrations and local monitoring pilots.
- Before deployment, it needs calibration records, protective mounting, and regular validation checks.
- Display-only systems still depend on correct sensor placement and trained users.

## Troubleshooting

| Problem | Possible Cause | Solution |
|---------|----------------|----------|
| The label is always stressed | One threshold is too strict or the gas sensor is still warming up | Allow more warm-up time and adjust the strongest threshold after observing the baseline. |
| The light reading never changes | The analog divider is wrong | Print the raw ADC value and rebuild the light-sensor path if needed. |
| Climate values stay at zero | The DHT22 read failed | Test the DHT22 separately and shorten the data wire if needed. |
| OLED is blank | The display library or I2C wiring is wrong | Confirm ssd1306.py is present and verify GPIO 20 and GPIO 21. |
