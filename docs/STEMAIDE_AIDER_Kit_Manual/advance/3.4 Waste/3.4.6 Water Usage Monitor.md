## Overview

The Pico counts pulses from a water-flow sensor and estimates both current flow rate and cumulative water usage.

Water conservation systems work better when users can see actual consumption instead of guessing how much water a fixture uses.

A Pico 2 W prototype with a pulse-output flow sensor and serial monitoring for liters per minute and total liters.

Interrupt-based pulse counting, calibration with a known volume, and careful handling of high-rate event data.

### Project Story

**Advanced Project**: This advanced project is designed to help learners move beyond basic wiring and coding into complete system thinking. The learner should build the prototype, test each subsystem, validate the data, explain the design decisions, and propose improvements for real-world deployment.

Water usage is difficult to improve without measurement. This project helps students connect raw pulse events to usable engineering quantities such as flow rate, total volume, and conservation planning.

## Testing Individual Components

Before running the full project, test each subsystem separately. This makes it easier to find wiring, library, or logic problems before full integration.

1. **Hardware setup**: Assemble the Pico, sensor, indicator, and load wiring exactly as shown in the connection table before applying power.
2. **Test the input sensor**: Count raw pulses first while water flows slowly so you know the interrupt is working.
3. **Test the output device**: No high-power actuator is used here, so test the serial output path by printing the pulse count at each interval.
4. **Test the decision logic**: Compare the calculated liters with a measured container fill so you can calibrate the pulse factor.
5. **Run the full system**: Run the full monitor through several sample windows and compare the live rate with the cumulative total.
6. **Validate the prototype**: Repeat the measurement at low and high flow rates to see whether the calibration factor stays stable.
7. **Save the project**: Save the validated program on the Pico as main.py and keep a copy on the computer for future edits.

## Full Project Code

After completing and checking the circuit connections, open Thonny IDE, copy and paste this code into a new file or upload the project file to the Raspberry Pi Pico 2 W, then run it from Thonny.

```python
from machine import Pin
import time

FLOW_PIN = 14
PULSES_PER_LITER = 450.0
SAMPLE_INTERVAL_S = 10

pulse_count = 0
total_pulses = 0

flow = Pin(FLOW_PIN, Pin.IN, Pin.PULL_UP)


def count_pulse(pin):
    global pulse_count, total_pulses
    pulse_count += 1
    total_pulses += 1


flow.irq(trigger=Pin.IRQ_FALLING, handler=count_pulse)
print("Water usage monitor ready")

while True:
    time.sleep(SAMPLE_INTERVAL_S)

    pulses = pulse_count
    pulse_count = 0

    liters_this_interval = pulses / PULSES_PER_LITER
    liters_per_min = liters_this_interval * (60 / SAMPLE_INTERVAL_S)
    total_liters = total_pulses / PULSES_PER_LITER

    print("Pulses: {} | Flow: {:.2f} L/min | Total: {:.2f} L".format(pulses, liters_per_min, total_liters))
```

## How the Code Works

| Code Section | What It Does | Why It Matters | What to Modify During Testing |
|--------------|--------------|----------------|------------------------------|
| Interrupt setup | Attaches an IRQ handler to the flow sensor pulse pin | Pulse counting is more reliable with interrupts than with slow polling loops | If the sensor output polarity differs, adjust the IRQ edge after checking the datasheet |
| Calibration constant | Defines how many pulses equal one liter | All volume calculations depend on this factor, so calibration matters | Replace PULSES_PER_LITER with the factor measured for your specific sensor |
| Sample window | Collects pulses for a fixed interval and converts them into liters per minute | A fixed sample window turns raw pulse counts into a meaningful rate | Change SAMPLE_INTERVAL_S after checking how responsive the display needs to be |
| Total volume tracking | Keeps a running total of all pulses received | Cumulative usage supports conservation analysis and comparison between tests | Reset total_pulses manually only when starting a new experiment |

## Expected Result

The serial monitor reports the current reading or state clearly.

The hardware output responds when the decision logic changes state.

Subsystem behavior matches the thresholds, timing, or rules described in the document.

### Validation Checks

- **Normal condition test**: run a small steady flow and confirm pulses increase during the sample window
- **Calibration test**: collect a known volume such as 1 liter and compare it with the reported total
- **Sensor reading validation**: compare slow flow and faster flow to see whether liters per minute change logically
- **False trigger test**: stop the flow completely and confirm pulses do not continue increasing
- **Limitation test**: explain that different sensors and pipe layouts can change the calibration factor

### Deployment and Limitations

- This prototype is useful for classroom analytics and local operational studies
- Before deployment, the sensor location and counting logic need validation against real traffic patterns
- Counts are only useful when users know the conditions that produce under-counting or over-counting

## Troubleshooting

| Problem | Possible Cause | Solution |
|---------|----------------|----------|
| No pulses are counted | Signal conditioning or wiring is wrong | Verify the signal level strategy and confirm the pulse line reaches GPIO 14 |
| Flow rate seems too high or too low | The pulse-per-liter factor is wrong | Measure a known water volume and recalculate PULSES_PER_LITER |
| Counts jump unexpectedly | Electrical noise or bouncing edges are present | Shorten the signal wire and check whether the sensor needs pull-up or filtering |
| Total liters resets unexpectedly | The board restarted or the code was rerun | Use a stable power source and note that this simple prototype does not persist totals across resets |
