## Overview

The Pico uses a reed switch to detect whether a door is open and raises an alert only if the door stays open too long.

Doors that remain open can create security, privacy, or sanitation problems, so the alert should depend on duration rather than a brief opening.

A Pico 2 W prototype with a reed switch, magnet, LED, and buzzer for door-open timing alerts.

Magnetic contact sensing, timing-based alerts, and simple occupancy-security logic.

## Step-by-Step Assembly

1. Mount the reed switch on the frame and the magnet on the moving door.
2. Connect one reed switch lead to GPIO 14 and the other to GND.
3. Connect the LED to GPIO 15 through a 220 ohm resistor and the LED cathode to GND.
4. Connect the buzzer positive pin to GPIO 16 and the negative pin to GND.
5. Open and close the door by hand while watching the raw switch state before enabling the timed alarm.

## Full Project Code

After completing and checking the circuit connections, open Thonny IDE, copy and paste this code into a new file or upload the project file to the Raspberry Pi Pico 2 W, then run it from Thonny.

```python
from machine import Pin
import time

REED_PIN = 14
LED_PIN = 15
BUZZER_PIN = 16
OPEN_ALERT_MS = 10000

reed = Pin(REED_PIN, Pin.IN, Pin.PULL_UP)
led = Pin(LED_PIN, Pin.OUT)
buzzer = Pin(BUZZER_PIN, Pin.OUT)
open_since_ms = None

print("Door open alert ready")

while True:
    door_open = reed.value() == 1
    now = time.ticks_ms()

    if door_open:
        if open_since_ms is None:
            open_since_ms = now
        open_duration = time.ticks_diff(now, open_since_ms)
        if open_duration >= OPEN_ALERT_MS:
            status = "ALERT"
            led.on()
            buzzer.on()
        else:
            status = "OPEN TIMING"
            led.off()
            buzzer.off()
    else:
        open_since_ms = None
        status = "CLOSED"
        led.off()
        buzzer.off()

    print("Status:", status)
    time.sleep_ms(500)
```

## How the Code Works

| Code Section | What It Does | Why It Matters | What to Modify During Testing |
|--------------|--------------|----------------|------------------------------|
| Switch setup | Defines the reed-switch input with an internal pull-up | Door sensors often need a known idle state so the code can interpret open and closed reliably | If your switch wiring uses the opposite polarity, invert the door_open test after measurement |
| Open timer | Starts timing when the door opens and resets when it closes | A timing threshold avoids nuisance alarms for normal quick openings | Adjust OPEN_ALERT_MS to match the real use case |
| Alert threshold logic | Separates OPEN TIMING from full ALERT mode | Students can validate the difference between brief and prolonged-open conditions | Change the threshold only after testing real door behavior |
| Outputs | Drives the LED and buzzer only during the prolonged-open alert | This keeps the system quiet during normal operation but visible during faults | If you want a softer warning stage, add LED-only logic before buzzer activation |

### Validation Checks

- **Normal condition test**: confirm the system stays in its safe or idle state under baseline conditions
- **Warning condition test**: move the input close to the chosen threshold and confirm the transition is sensible
- **Critical condition test**: trigger the strongest alert or control state and confirm the output response is correct
- **Calibration test**: compare the sensor or timing against a known reference or repeated trial
- **Limitation test**: deliberately create an awkward or noisy condition and note how the prototype behaves

### Deployment and Limitations

- This prototype can be used for local warning demonstrations and early field trials
- Before deployment, it needs enclosure protection, calibration records, and a clear response procedure
- Alert systems should always be treated as decision-support tools unless they are certified for the application

## Troubleshooting

| Problem | Possible Cause | Solution |
|---------|----------------|----------|
| Door state is reversed | Reed-switch polarity assumption is wrong | Print the raw input state and invert the door_open condition if needed |
| Alert never starts | Threshold is too long or the door does not stay open long enough | Lower OPEN_ALERT_MS for a quick validation test |
| Door appears open even when closed | Magnet alignment is poor | Reposition the magnet so it sits close to the reed switch when the door shuts |
| Buzzer never sounds | Buzzer wiring is wrong or the alert duration is never reached | Test the buzzer directly and then hold the door open beyond the threshold |
