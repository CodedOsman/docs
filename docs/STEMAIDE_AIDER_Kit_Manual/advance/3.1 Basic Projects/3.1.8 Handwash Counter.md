## Step-by-Step Assembly

1. Connect the IR sensor to Pico 3.3V, GND, and GPIO 14.
2. Connect the LED to GPIO 15 through a 220 ohm resistor and connect the LED cathode to GND.
3. Aim the sensor so a hand passes through the beam once during a normal wash action.

## Full Project Code

After completing and checking the circuit connections, open Thonny IDE, copy and paste this code into a new file or upload the project file to the Raspberry Pi Pico 2 W, then run it from Thonny.

```python
from machine import Pin
import time

IR_PIN = 14
LED_PIN = 15
LOCKOUT_MS = 3000

ir = Pin(IR_PIN, Pin.IN, Pin.PULL_UP)
led = Pin(LED_PIN, Pin.OUT)
last_state = 1
last_count_ms = -LOCKOUT_MS
count = 0

print("Handwash counter ready")

while True:
    now = time.ticks_ms()
    current_state = ir.value()
    new_detection = last_state == 1 and current_state == 0

    if new_detection and time.ticks_diff(now, last_count_ms) >= LOCKOUT_MS:
        count += 1
        last_count_ms = now
        led.on()
        time.sleep_ms(200)
        led.off()
        print("Handwash count:", count)

    last_state = current_state
    time.sleep_ms(50)
```

## How the Code Works

| Code Section | What It Does | Why It Matters | What to Modify During Testing |
|--------------|--------------|----------------|------------------------------|
| Input and output setup | Defines the IR sensor input and the LED output | This connects one event source to one clear visible response | Confirm whether your sensor idles high or low before changing the logic |
| Edge detection | Counts only a new active-low detection event | Edge detection prevents one long detection from becoming many counts | Do not replace edge logic with a simple level check unless you want repeated counts |
| Lockout timing | Ignores new counts during a short cooldown period | A lockout helps reduce over-counting when users linger in front of the sensor | Tune LOCKOUT_MS after watching real user behavior |
| Count feedback | Flashes the LED and prints the new total | Immediate feedback makes validation easier during live testing | Change the LED flash duration only after the logic is stable |

### Validation Checks

- **Normal condition test**: confirm the system stays in its safe or idle state under baseline conditions
- **Warning condition test**: move the input close to the chosen threshold and confirm the transition is sensible
- **Critical condition test**: trigger the strongest alert or control state and confirm the output response is correct
- **Calibration test**: compare the sensor or timing against a known reference or repeated trial
- **Limitation test**: deliberately create an awkward or noisy condition and note how the prototype behaves

### Deployment and Limitations

- This prototype is useful for classroom analytics and local operational studies
- Before deployment, the sensor location and counting logic need validation against real traffic patterns
- Counts are only useful when users know the conditions that produce under-counting or over-counting

## Troubleshooting

| Problem | Possible Cause | Solution |
|---------|----------------|----------|
| Count increases too fast | Lockout is too short or the sensor sees repeated motion | Increase LOCKOUT_MS and adjust the sensor angle |
| No counts appear | Sensor output is not reaching GPIO 14 | Print the raw sensor state and recheck the signal wire |
| LED never flashes | LED wiring or resistor path is wrong | Test the LED separately from the counter logic |
| Counts do not match real visits | Sensor placement does not match actual hand movement | Reposition the sensor so one handwash motion crosses the detection area clearly |
