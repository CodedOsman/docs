## Overview

The Pico reads an SOS button, tries to capture a GPS fix, and sends a local Wi-Fi alert message when the user requests help.

Field workers, sanitation staff, or student teams may need a simple way to raise a location-aware alert during a safety incident.

A Pico 2 W prototype with a button, status LED, NEO-6M style GPS module, and Wi-Fi alert transmission to a local endpoint.

Event triggering, fallback behavior when GPS or Wi-Fi fail, and honest local-network alerting rather than a finished emergency platform.

### Project-Specific Setup Notes

- No external library is required. This project uses built-in MicroPython modules such as network, socket, and ujson.
- Enter your 2.4 GHz Wi-Fi network name and password in the code before testing the communication step.
- Set the local server IP address, port, and path in the code if you want the prototype to send data to a school or lab endpoint.
- This project demonstrates local sensing and data exchange only. It does not create a complete cloud platform by itself.
- The code sends HTTP data to a local network endpoint. It does not contact a real emergency service and should be treated as a prototype only.

### Project-Specific Safety Note

Check that the GPS module UART output is 3.3V logic. If the module uses 5V TTL, add a level shifter or voltage divider.

Do not rely on this prototype for real emergency response. Validate communication with a local test server only.

## Circuit Connections

|---------------|-------------|---------------------------------|-------|
| SOS button leg 1 | GPIO 14 | GPIO 14 / physical pin 19 | Use with an internal pull-down or pull-up design |
| SOS button leg 2 | 3.3V output | Physical pin 36 | High when pressed in this design |
| LED anode | 220 ohm resistor to GPIO 15 | GPIO 15 / physical pin 20 | Status LED |
| LED cathode | GND | Physical pin 23 or 38 | Ground return |
| GPS VCC | Module-rated supply | Use the module label | Many GPS boards accept 3.3V or 5V input |
| GPS GND | GND | Physical pin 38 | Common ground |
| GPS TX | GPIO 5 UART1 RX | GPIO 5 / physical pin 7 | Main NMEA data path to Pico |
| GPS RX | GPIO 4 UART1 TX | GPIO 4 / physical pin 6 | Optional if you do not send GPS configuration |

---

## Wiring Diagram

```
  Button               -> GPIO 14 and 3.3V
  GPIO 15              -> 220 ohm resistor -> LED anode
  GPS TX               -> GPIO 5
  GPIO 4               -> GPS RX (optional)
  Common GND           -> LED and GPS
  Wi-Fi uses the Pico 2 W radio and needs no extra wiring
```

## Step-by-Step Assembly

1. Wire the SOS button between GPIO 14 and 3.3V so the pin reads high when pressed.
2. Connect the LED to GPIO 15 through a 220 ohm resistor and connect the LED cathode to GND.
3. Connect the GPS module TX line to GPIO 5 and its GND to Pico GND.
4. Connect GPS RX to GPIO 4 only if your module setup requires it.
5. Power the GPS module from the voltage recommended by its board label and confirm the UART signal is 3.3V-safe.

## Testing Individual Components

Before running the full project, test each subsystem separately. This makes it easier to find wiring, library, or logic problems before full integration.

1. **Hardware setup**: Assemble the Pico, sensor, indicator, and load wiring exactly as shown in the connection table before applying power.
2. **Test the input sensor**: Print button state changes and GPS sentences separately so you can validate both inputs before using Wi-Fi.
3. **Test the output device**: Blink the LED with a short test script so the user feedback path is known to work.
4. **Test the decision logic**: Press the button with Wi-Fi disabled and confirm the code still creates a local alert payload instead of failing silently.
5. **Run the full system**: Connect to a local HTTP test server and confirm the alert payload is received when the button is pressed.
6. **Validate the prototype**: Test indoors and outdoors to compare how often the GPS produces a valid location fix.
7. **Save the project**: Save the validated program on the Pico as main.py and keep a copy on the computer for future edits.

## Full Project Code

After completing and checking the circuit connections, open Thonny IDE, copy and paste this code into a new file or upload the project file to the Raspberry Pi Pico 2 W, then run it from Thonny.

```python
import network
import socket
import time
import ujson as json
from machine import Pin, UART

BUTTON_PIN = 14
LED_PIN = 15
GPS_TX_PIN = 4
GPS_RX_PIN = 5
ALERT_COOLDOWN_MS = 10000
WIFI_SSID = "YOUR_WIFI"
WIFI_PASSWORD = "YOUR_PASSWORD"
SERVER_HOST = "192.168.1.50"
SERVER_PORT = 8000
SERVER_PATH = "/alerts"

button = Pin(BUTTON_PIN, Pin.IN, Pin.PULL_DOWN)
led = Pin(LED_PIN, Pin.OUT)
gps = UART(1, baudrate=9600, tx=Pin(GPS_TX_PIN), rx=Pin(GPS_RX_PIN))
last_button_state = 0
last_alert_ms = -ALERT_COOLDOWN_MS


def connect_wifi(timeout_s=15):
    wlan = network.WLAN(network.STA_IF)
    if wlan.isconnected():
        return wlan

    wlan.active(True)
    wlan.connect(WIFI_SSID, WIFI_PASSWORD)

    for _ in range(timeout_s):
        if wlan.isconnected():
            return wlan
        time.sleep(1)

    return None


def nmea_to_decimal(value, direction):
    if not value:
        return None
    dot_index = value.find(".")
    if dot_index < 0:
        return None

    degree_length = dot_index - 2
    degrees = int(value[:degree_length])
    minutes = float(value[degree_length:])
    decimal = degrees + minutes / 60
    if direction in ("S", "W"):
        decimal = -decimal
    return round(decimal, 6)


def read_gps_fix(timeout_ms=3000):
    start = time.ticks_ms()
    while time.ticks_diff(time.ticks_ms(), start) < timeout_ms:
        line = gps.readline()
        if not line:
            time.sleep_ms(100)
            continue

        try:
            text = line.decode().strip()
        except UnicodeError:
            continue

        if text.startswith("$GPRMC") or text.startswith("$GNRMC"):
            parts = text.split(",")
            if len(parts) > 6 and parts[2] == "A":
                latitude = nmea_to_decimal(parts[3], parts[4])
                longitude = nmea_to_decimal(parts[5], parts[6])
                return {"lat": latitude, "lon": longitude}
    return None


def send_alert(payload):
    address = socket.getaddrinfo(SERVER_HOST, SERVER_PORT)[0][-1]
    body = json.dumps(payload)
    request = "POST {} HTTP/1.1\r\nHost: {}\r\nContent-Type: application/json\r\nContent-Length: {}\r\nConnection: close\r\n\r\n{}".format(SERVER_PATH, SERVER_HOST, len(body), body)

    sock = socket.socket()
    try:
        sock.connect(address)
        sock.send(request.encode())
        sock.recv(128)
        return True
    finally:
        sock.close()


def blink(times):
    for _ in range(times):
        led.on()
        time.sleep_ms(150)
        led.off()
        time.sleep_ms(150)


print("Wearable alert prototype ready")

while True:
    now = time.ticks_ms()
    button_state = button.value()
    new_press = button_state == 1 and last_button_state == 0

    if new_press and time.ticks_diff(now, last_alert_ms) >= ALERT_COOLDOWN_MS:
        gps_fix = read_gps_fix()
        payload = {
            "device": "wearable_255",
            "event": "SOS",
            "uptime_s": time.ticks_ms() // 1000,
            "gps": gps_fix,
        }

        wlan = connect_wifi()
        if wlan is not None:
            success = send_alert(payload)
        else:
            success = False

        print("Alert payload:", payload)
        print("Alert sent" if success else "Alert stored locally only")
        blink(2 if success else 5)
        last_alert_ms = now

    last_button_state = button_state
    time.sleep_ms(50)
```

## How the Code Works

| Code Section | What It Does | Why It Matters | What to Modify During Testing |
|--------------|--------------|----------------|------------------------------|
| Input setup | Configures the SOS button, LED, and UART GPS module | The project depends on both user input and location data working together | Verify button wiring and GPS UART pins before testing Wi-Fi |
| GPS parsing | Converts NMEA latitude and longitude into decimal coordinates when a valid fix is available | Students can see that location data is often incomplete or delayed in real systems | Increase read_gps_fix timeout if your module needs longer to deliver a fix |
| Wi-Fi and HTTP send | Connects to the local network and posts the alert payload to a test endpoint | This turns the project into a communication prototype instead of a serial-only demo | Change the server host, port, and path to match your local test service |
| Alert fallback | Keeps the payload visible in the Shell even when GPS or Wi-Fi is missing | A prototype should fail visibly instead of pretending the alert worked | Adjust the LED blink pattern if you want different feedback for success and failure |

### Validation Checks

- **Normal condition test**: power the device and confirm no alert is sent until the button is pressed
- **Sensor reading validation**: confirm that the button press is detected once and that GPS data appears when a fix is available
- **Communication validation**: connect to a local test server and confirm that one press produces one HTTP alert payload
- **Fallback test**: disable Wi-Fi and confirm the code still prints the unsent payload locally
- **Limitation test**: compare indoor and outdoor GPS performance and record how long a valid fix takes

### Deployment and Limitations

- This prototype is suitable for local pilots where sensor data must be logged or transmitted over short periods
- Before deployment, network reliability, power backup, and endpoint security need attention
- The system should keep useful local behavior even when communication fails

## Troubleshooting

| Problem | Possible Cause | Solution |
|---------|----------------|----------|
| No GPS coordinates appear | Module has no sky view or UART pins are wrong | Test the GPS outdoors and confirm GPS TX reaches Pico GPIO 5 |
| Button presses do nothing | Button wiring or cooldown logic is wrong | Check that the button really drives GPIO 14 high when pressed |
| Wi-Fi never connects | SSID, password, or 2.4 GHz network settings are wrong | Confirm the credentials and test the Pico W with a simpler Wi-Fi example first |
| Server never receives alerts | HTTP host, port, or endpoint path is incorrect | Run a local test server and verify the destination values in the code |
