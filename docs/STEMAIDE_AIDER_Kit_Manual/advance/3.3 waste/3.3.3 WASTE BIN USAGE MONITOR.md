## Overview

The Pico counts discrete usage events with an IR or break-beam style sensor and reports the total locally.

Operational planning is difficult when usage is estimated instead of counted.

A Pico 2 W prototype with an IR detection sensor, status LED, and event counter logic.

Edge-based counting, false-trigger reduction, and simple operational data interpretation.

### Project Story

**Advanced Project**: This advanced project is designed to help learners move beyond basic wiring and coding into complete system thinking. The learner should build the prototype, test each subsystem, validate the data, explain the design decisions, and propose improvements for real-world deployment.

Event counters can support planning for waste-bin usage monitoring when the sensor placement and debounce logic are validated carefully.

## Step-by-Step Assembly

1. Connect the IR sensor output to GPIO 14 and power the module according to its rating.

## Full Project Code

After completing and checking the circuit connections, open Thonny IDE, copy and paste this code into a new file or upload the project file to the Raspberry Pi Pico 2 W, then run it from Thonny.

```python
from machine import Pin
import time

SENSOR_PIN = 14
LED_PIN = 15
ACTIVE_STATE = 0
LOCKOUT_SECONDS = 1.0

sensor = Pin(SENSOR_PIN, Pin.IN, Pin.PULL_UP)
led = Pin(LED_PIN, Pin.OUT)
count = 0
last_trigger_time = 0
active_before = False

print('IR counter ready')

while True:
    active_now = sensor.value() == ACTIVE_STATE
    now = time.time()

    if active_now and not active_before and now - last_trigger_time >= LOCKOUT_SECONDS:
        count += 1
        last_trigger_time = now
        led.value(1)
        print('Count:{}'.format(count))
    else:
        led.value(0)

    active_before = active_now
    time.sleep(0.05)
```

## How the Code Works

| Code Section | What It Does | Why It Matters | What to Modify During Testing |
|--------------|--------------|----------------|------------------------------|
| Edge detection | Counts only on the transition into the active state | This prevents one long detection from being counted many times | Confirm the sensor active state before trusting the count |
| Lockout timing | Ignores extra triggers for a short time after each event | Lockout helps reduce false double counts | Increase the lockout if events are still counted twice |
| Event flash | Turns on the LED briefly for each counted event | The flash gives immediate confirmation that the event was registered | If the LED feels too fast, add a short pulse hold during testing |

## Expected Result

The serial monitor reports the current reading or state clearly.

Subsystem behavior matches the thresholds, timing, or rules described in the document.

### Validation Checks

- **Normal condition test**: confirm the system stays in its safe or idle state under baseline conditions
- **Warning condition test**: move the input close to the chosen threshold and confirm the transition is sensible
- **Critical condition test**: trigger the strongest alert or control state and confirm the output response is correct
- **Calibration test**: compare the sensor or timing against a known reference or repeated trial
- **Limitation test**: deliberately create an awkward or noisy condition and note how the prototype behaves

### Deployment and Limitations

- This prototype is useful for classroom analytics and local operational studies
- Before deployment, the sensor location and counting logic need validation against real traffic patterns
- Counts are only useful when users know the conditions that produce under-counting or over-counting

## Troubleshooting

| Problem | Possible Cause | Solution |
|---------|----------------|----------|
| The count rises too fast | The sensor is seeing repeated reflections or the lockout is too short | Reposition the sensor and increase the lockout time |
| No events are counted | The active state is reversed or the sensor is misaligned | Print the raw pin value and adjust the active-state constant if needed |
| The LED never flashes | The LED wiring is wrong or events are not being detected | Test GPIO 15 with a blink script and then recheck the sensing path |
| One object counts multiple times | The detection zone is too wide or unstable | Narrow the sensor aim and add more lockout time |
