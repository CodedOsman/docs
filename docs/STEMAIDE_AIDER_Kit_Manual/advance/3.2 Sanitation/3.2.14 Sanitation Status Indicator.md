Simple visible status tools help users know whether a sanitation space is available or needs attention.

A Pico 2 W prototype with one push button and three colored LEDs representing different sanitation states.

State transitions, user feedback, and practical discussion of human-readable system status.

## Circuit Connections

|---------------|-------------|---------------------------------|-------|
| Button leg 1 | GPIO 14 | GPIO 14 / physical pin 19 | Use internal pull-down |
| Button leg 2 | 3.3V output | Physical pin 36 | Button drives the pin high when pressed |
| Green LED anode | 220 ohm resistor to GPIO 15 | GPIO 15 / physical pin 20 | CLEAN state |
| Yellow LED anode | 220 ohm resistor to GPIO 16 | GPIO 16 / physical pin 21 | IN USE state |
| Red LED anode | 220 ohm resistor to GPIO 17 | GPIO 17 / physical pin 22 | NEEDS SERVICE state |
| All LED cathodes | GND | Physical pin 18, 23, or 38 | Ground return |

## Wiring Diagram

```
  Button -> GPIO 14 and 3.3V
  GPIO 15 -> resistor -> green LED anode
  GPIO 16 -> resistor -> yellow LED anode
  GPIO 17 -> resistor -> red LED anode
  All LED cathodes -> GND
```

3. Connect the yellow LED to GPIO 16 through a 220 ohm resistor and connect its cathode to GND.
4. Connect the red LED to GPIO 17 through a 220 ohm resistor and connect its cathode to GND.

## Testing Individual Components

Before running the full project, test each subsystem separately. This makes it easier to find wiring, library, or logic problems before full integration.

1. **Hardware setup**: Assemble the Pico, sensor, indicator, and load wiring exactly as shown in the connection table before applying power.
2. **Test the input sensor**: Print the button state and confirm it changes cleanly between released and pressed.
3. **Test the output device**: Test each LED separately before linking them to the status logic.
4. **Test the decision logic**: Confirm that short presses cycle the states and a long press resets the system to CLEAN.
5. **Run the full system**: Run a sequence of repeated presses and confirm that the LEDs and printed state always match.
6. **Validate the prototype**: Ask another student to interpret the LEDs without explanation and note whether the meanings are obvious.
7. **Save the project**: Save the validated program on the Pico as main.py and keep a copy on the computer for future edits.

## Full Project Code

After completing and checking the circuit connections, open Thonny IDE, copy and paste this code into a new file or upload the project file to the Raspberry Pi Pico 2 W, then run it from Thonny.

```python
from machine import Pin
import time

BUTTON_PIN = 14
GREEN_PIN = 15
YELLOW_PIN = 16
RED_PIN = 17
LONG_PRESS_MS = 1500

button = Pin(BUTTON_PIN, Pin.IN, Pin.PULL_DOWN)
green = Pin(GREEN_PIN, Pin.OUT)
yellow = Pin(YELLOW_PIN, Pin.OUT)
red = Pin(RED_PIN, Pin.OUT)

states = ["CLEAN", "IN USE", "NEEDS SERVICE"]
state_index = 0
last_state = 0
press_start = 0


def show_state(index):
    green.value(1 if index == 0 else 0)
    yellow.value(1 if index == 1 else 0)
    red.value(1 if index == 2 else 0)
    print("Status:", states[index])


show_state(state_index)
print("Sanitation status indicator ready")

while True:
    now = time.ticks_ms()
    current = button.value()

    if current == 1 and last_state == 0:
        press_start = now
    elif current == 0 and last_state == 1:
        press_length = time.ticks_diff(now, press_start)
        if press_length >= LONG_PRESS_MS:
            state_index = 0
        else:
            state_index = (state_index + 1) % len(states)
        show_state(state_index)

    last_state = current
    time.sleep_ms(50)
```

## How the Code Works

| Code Section | What It Does | Why It Matters | What to Modify During Testing |
|--------------|--------------|----------------|------------------------------|

| State list | Stores the ordered list of sanitation states | Consistent state naming matters for user trust | Only change the order if the user workflow changes too |

### Validation Checks

- **Normal condition test**: confirm the system stays in its safe or idle state under baseline conditions
- **Warning condition test**: move the input close to the chosen threshold and confirm the transition is sensible
- **Critical condition test**: trigger the strongest alert or control state and confirm the output response is correct
- **Calibration test**: compare the sensor or timing against a known reference or repeated trial
- **Limitation test**: deliberately create an awkward or noisy condition and note how the prototype behaves

### Deployment and Limitations

- This prototype is useful for local indicator boards and classroom control panels
- Before deployment, labeling, enclosure design, and user training need improvement
- Status tools only work well when users trust the meaning of each state and keep the system updated

## Troubleshooting

| Problem | Possible Cause | Solution |
|---------|----------------|----------|
| States skip too quickly | Button bounce or long press confusion is occurring | Keep the debounce delay and adjust LONG_PRESS_MS after real testing |
| Wrong LED lights up | LED wiring does not match the intended state color | Recheck the GPIO mapping for each colored LED |
| Button does nothing | Button wiring is wrong or the pin never goes high | Print the button state and confirm the button is wired between 3.3V and GPIO 14 |
| Users misunderstand the board | The color meaning is not obvious | Add labels, icons, or training notes to the prototype board |
