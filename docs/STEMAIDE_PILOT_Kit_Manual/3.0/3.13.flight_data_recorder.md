**Student & Teacher Manual**

**1. Project Overview**

Every commercial aircraft carries a Flight Data Recorder — the 'black box' — that continuously logs hundreds of flight parameters to crash-resistant storage. When an accident occurs, investigators download this data and reconstruct every moment of the flight. This project builds a functional miniature FDR using an ESP32, BMP280, MPU6050, and MicroSD card, installs it on the Project 11 RC aircraft, and records a real flight to CSV. After landing, students download, graph, and analyse the data — exactly as accident investigators do with a real FDR.

**Learning Objectives**

* Describe the five key features of a real Solid State Flight Data Recorder (SSFDR): what it records, sample rate, storage medium, crash protection, and read-out method
* Wire a BMP280, MPU6050, and MicroSD module to an ESP32 and confirm all three are functional
* Write a data logging sketch that saves altitude, pitch, roll, G-force, and temperature to a CSV file at 10 Hz
* Install the FDR on the Project 11 RC aircraft and record a complete flight
* Import the CSV into a spreadsheet; graph altitude, pitch, and G-force vs. time; identify all flight phases
* Identify a hard landing event as a G-force spike in the recorded data

|  |  |

**Lessons 3–4 – Flight, Download & Analysis**

|  |  |
| --- | --- |
| **STEP 5** | **Record a Flight** |
|  | * Power on FDR at least 60 seconds before takeoff * Fly a complete circuit: takeoff → climb → level cruise → descend → landing * Include at least one deliberate hard landing (teacher supervised) to create a detectable G-force event * Power off immediately after landing; remove SD card; copy CSV to computer |

|  |  |
| --- | --- |
| **STEP 6** | **Data Analysis & Graph** |
|  | * Open CSV in spreadsheet; inspect the first 10 rows for any anomalies * Create Chart 1: Time (x) vs. Altitude (m) — label takeoff, cruise, descent, landing * Create Chart 2: Time (x) vs. Pitch (°) — identify nose-up on takeoff; level cruise; nose-down descent * Create Chart 3: Time (x) vs. AccZ (g) — identify the hard landing spike (AccZ > 2g) * Write a 1-page accident investigation report: describe what the data shows happened during the flight |

**5. Engineering Principles**

**FDR Parameter Comparison**

| **Parameter** | **Real Aircraft FDR** | **This Project's FDR** | **Sample Rate** |
| --- | --- | --- | --- |
| **Altitude** | Barometric + GPS | BMP280 barometric | 2 Hz |
| **Pitch & Roll** | Gyro + Accelerometer | MPU6050 (complementary filter) | 10 Hz |
| **G-Force** | 3-axis accelerometer | MPU6050 AccX/Y/Z | 10 Hz |
| **Temperature** | OAT probe | BMP280 temperature | 1 Hz |
| **Timestamp** | UTC from GPS | ESP32 millis() counter | Every record |
| **Storage** | Crash-resistant SSFDR | MicroSD card (FAT32) | Continuous |
| **Playback** | DFDR replay system | CSV in Excel/LibreOffice | Post-flight |

|  |
| --- |
| **Real FDR – How It Works**  A modern Solid State Flight Data Recorder (SSFDR) records 25+ hours of flight data to crash-resistant NAND flash memory.  ICAO Annex 6 mandates FDRs on all commercial aircraft above 5,700 kg maximum takeoff weight.  The recorder must survive: 3,400 g impact; 1100°C fire for 30 minutes; 6,000 m submersion for 30 days.  When investigators retrieve the FDR, they replay the data using Digital Flight Data Replay software to reconstruct the entire flight in 3D animation.  This project records the same parameters — altitude, attitude, G-force, temperature — at a lower sample rate and without crash protection. |

**8. Common Errors & Fixes**

| **Error** | **Likely Cause** | **Fix** |
| --- | --- | --- |
| **SD card not initialised** | Card not FAT32 or CS pin wrong | Format as FAT32; verify CS pin (GPIO 5) connection |
| **File created but empty** | SD write failing silently | Add Serial.println after each SD write; check error codes from SD library |
| **Altitude data noisy** | BMP280 no filter applied | Enable IIR filter in BMP280 library config; add software moving average |
| **Pitch and roll drifting** | No complementary filter | Apply filter from Project 48: angle = 0.98×(gyro)+0.02×(accel) |
| **Too large a file (hundreds of MB)** | Sample rate too high | Reduce to 10 Hz for motion data; 2 Hz for altitude; 1 Hz for temperature |
| **Data missing during landing** | Power cut before file closed | Add file.close() call triggered by low throttle or manual switch |

**9. Upgrade & Extension Ideas**

* GPS Timestamp: add the NEO-6M GPS module from Project 9; replace millis() with UTC time for proper time-stamping
* Multiple Flights: modify code to create FLIGHT001.CSV, FLIGHT002.CSV etc. on each power cycle; compare two flights side-by-side
* Crash Simulation: intentionally tip the aircraft over on landing; identify the event signature in the accelerometer data
* Web Dashboard: use the ESP32 Wi-Fi to serve the CSV data as a live web page viewable on a phone

**10. Teacher Notes & Differentiation**

* The accident investigation report (Step 6) is the most powerful deliverable — frame it as a real investigation scenario: 'The aircraft was found at the edge of the airfield. The pilot cannot remember what happened. Analyse the FDR data and write your report'
* Show a real accident investigation report (freely available from GCAA or NTSB website) before students write theirs — the format and language are informative
* Support – Pre-wired FDR unit; teacher installs on aircraft; students focus on CSV analysis and graph production only
* Core – Full build; aircraft installation; flight; CSV analysis; three graphs; investigation report
* Extension – GPS timestamp; multiple flights; web dashboard; crash simulation analysis
